feat(integrations): humanize Connection errors + dashboard pill flips on test failure

Three discrete pieces in this commit, all surfaced by hands-on
debugging of the dev WP / api network-namespace mismatch on this
machine. None are speculative.

================================================================
1. Humanize transient connect failures
================================================================

apps/api/src/server/routers/integrations.ts
  - test mutation now maps `kind: "transient" | "timeout"` adapter
    results into a single `Connection error — cannot reach <host>:
    <raw>` string instead of leaking the raw `woo network error:
    probeWooReachability /wp-json: fetch failed: connect
    ECONNREFUSED 127.0.0.1:8881` to the merchant-facing modal.
    Auth/schema kinds keep their exact wording so credential vs.
    connectivity stay separable in the runbook.
  - On any failed test it now ALSO writes top-level
    `lastSyncStatus = "error"`, `lastError = friendlyDetail`, and
    bumps `errorCount`. Without this the dashboard's `getHealth`
    reader projects `lastSyncStatus` for the pill, but the test
    mutation only ever wrote `health.lastError` (a different
    field) — so the pill stayed green/idle until the next
    scheduled sync stamped the row. Now the pill flips red within
    one Test click.
  - On a successful test it resets all three back to ok / null /
    0 so a flaky transient that bumped `errorCount` once decays
    back to zero on the next green run.

================================================================
2. UI surfaces "Connection error" with merchant-actionable copy
================================================================

apps/web/src/components/integrations/IntegrationHealthCard.tsx
  - When `status === "error"` AND `lastError` matches connection-
    class signals (Connection error / ECONNREFUSED / ENOTFOUND /
    timeout / network error), render a dedicated red inline alert
    next to the existing webhookSilent / unhealthy / disabled
    flags. Copy: "Connection error — your store appears
    unreachable. Check that the URL is correct and the server is
    accepting traffic, then click Test connection."
  - Auth-rejected and schema-drift errors keep going through the
    generic `lastError` block — different runbook, different copy.

================================================================
3. Build chain unblocked
================================================================

apps/api/tsconfig.build.json
  - paths repointed at `packages/<name>/dist/index.d.ts` (the
    actual emitted artifacts) instead of the previous
    `packages/<name>/src/index.d.ts` (which never existed in
    src/, only in dist/). The dangling path triggered a TSC
    `Debug Failure: resolveExternalModule` on every `npm run
    build` — the same one that left `apps/api/dist/` partially
    populated. The build now resolves cleanly and emits all of
    `apps/api/src/**` to `apps/api/dist/**` in one shot.

================================================================
4. End-to-end visibility
================================================================

apps/web/e2e/shopify-import-progress.spec.ts  (new)
  - Playwright spec that asserts the import modal NEVER shows
    `imported = 0 AND failed = 0` simultaneously when Shopify
    returned at least one order. Walks the full merchant flow:
    sign up -> connect (custom-app credentials path) -> click
    Import recent -> wait for terminal state -> read each tile.
    Pass condition: imported + duplicates + failed > 0  OR  the
    `lastError` block is rendered. Either way the merchant gets
    actionable signal — not the silent "everything zero" mystery
    the dashboard used to show on a phone-less batch.
