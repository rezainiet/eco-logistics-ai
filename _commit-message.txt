security(auth): harden every auth form against native GET fallback

Saw this in dev stdout while debugging an unrelated Shopify install:

    GET /Login?email=masudreza%40gmail.com&password=Masud2003 200 in 325ms

If React hasn't hydrated when the user clicks Submit (slow first
compile, hydration error, JS disabled), the browser falls back to a
native form submission. The default method is GET, which serializes
every named field into the URL — credentials end up in the URL bar,
browser history, every reverse-proxy access log, and our own dev
server stdout.

Same defence-in-depth applied to all five auth-adjacent forms so the
pattern is uniform and easy to audit:

  1. method="post" — even on native fallback, no fields go in the URL
  2. action="/api/auth/__nope" — non-existent endpoint, so a native
     fall-through gets a loud 404 instead of silently reloading with
     credentials still in memory
  3. onSubmit's first line is e.preventDefault() — runs unconditionally,
     before react-hook-form's handleSubmit can succeed or throw

Forms hardened:
  - apps/web/src/app/(auth)/login/page.tsx
  - apps/web/src/app/(auth)/signup/page.tsx
  - apps/web/src/app/reset-password/page.tsx
  - apps/web/src/app/forgot-password/page.tsx (lower-stakes — only
    takes an email — but applied for consistency so there's no
    exception that could miss a future field)
  - apps/web/src/app/dashboard/settings/page.tsx (change-password form)

Note for whoever rotates secrets: the password "Masud2003" appeared
in dev stdout (and in screenshots taken during the debug session), so
the local dev account credentials should be rotated.
