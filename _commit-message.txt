fix(shopify-oauth): unblock pending integrations stuck in canonicalize loop

Three compounding bugs caused real merchant Shopify installs to land
on Shopify's side but stay `pending` forever in our dashboard. Saw it
in real logs:

    [shopify-oauth] callback received (no installStartedAt) { shop: 'dwykhp-en...' }
    [shopify-oauth] canonicalizing accountKey { from: 'devs-9807...', to: 'dwykhp-en...' }
    ...same two lines repeating with no transition to `connected`...

Root causes:

1. `installStartedAt` was missing from the Integration schema.
   Mongoose strict mode silently dropped every write of the dot-set
   `credentials.installStartedAt`, so every callback logged
   "(no installStartedAt)" and we lost the single best signal for
   diagnosing slow installs ("is it Shopify or us?"). The whole
   long-install timing UX we built was running blind.

2. The connect mutation did the install nonce + timestamp in a
   SECOND `updateOne` after the upsert, using dot-notation. That was
   broken in two ways: bug #1 above made `installStartedAt` a no-op,
   and the brief gap between writes was a real race window where a
   callback could land on a row with no nonce → "integration_not_found".
   Refactored to a single atomic upsert that includes nonce +
   installStartedAt in the same write.

3. THE BIG ONE — canonicalization save crashed with E11000.
   When Shopify rewrites the shop hostname (vanity → canonical, e.g.
   `devs-9807` → `dwykhp-en`), the callback updates `accountKey` and
   saves. But the unique index `(merchantId, provider, accountKey)`
   refused the write whenever a stale `disconnected` row from a
   previous install/uninstall cycle already held the canonical key.
   The exception fell through unhandled, the row stayed `pending`,
   the merchant clicked Connect again, and we looped forever.

   Fixed in two layers:
   - Before re-keying, `Integration.deleteMany` for any
     status:"disconnected" row holding the canonical accountKey for
     this merchant + provider. Connected rows are NEVER touched —
     we'd rather E11000 than silently overwrite a working integration.
   - Wrapped `integration.save()` in try/catch. Any future
     constraint surprise lands as a friendly
     `?error=callback_save_failed` redirect with an audit-log entry,
     not a hung tab.

Files:
  - packages/db/src/models/integration.ts        (+7 lines, schema)
  - apps/api/src/server/routers/integrations.ts  (atomic upsert)
  - apps/api/src/server/webhooks/integrations.ts (orphan cleanup + save try/catch)

---

Bonus from the same debugging session:

- apps/web/src/app/dashboard/integrations/page.tsx — added
  aria-label="Disconnect integration" + matching title to the
  icon-only trash button on each Connections row. Without it the
  button has no accessible name (screen readers say just "button"
  with no context). Also makes Playwright role-based queries find it.

- apps/web/e2e/shopify-connect.spec.ts — new Playwright spec
  covering the merchant-facing connect surface end-to-end:
  signup → open dialog → domain validation → fill Advanced creds →
  integration appears connected → admin-bump to Scale tier →
  one-shop-per-provider UI guard fires (card flips to disabled
  "Connected") → disconnect → reconnect different shop succeeds.
  The OAuth callback path itself isn't covered (would require
  intercepting outbound HTTPS to Shopify) — covered by API
  integration tests instead.
